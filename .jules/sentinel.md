## 2024-05-18 - [Use of GET Request Method With Sensitive Query Strings]
**Vulnerability:** A password to access admin reports was sent in the query parameter (CWE-598).
**Learning:** Query parameters are frequently logged by web servers, proxies, and browser history, exposing sensitive authentication tokens. The frontend was constructed to use a simple URL string interpolation rather than securely transmitting credentials.
**Prevention:** Avoid putting credentials or sensitive tokens in the query string or URL path for any HTTP method. For sensitive endpoints, consider migrating to POST requests using JSON body authentication, or use standard Authorization headers for GET requests.
## 2024-05-30 - Prevent Authorization Bypasses and Identity Spoofing in Koa Middleware
**Vulnerability:** The custom Koa middleware `requireSignature` used exact path matching for protected routes instead of broad prefixes, which left sub-routes (like nested marketplace or messages routes) exposed to unauthorized access. In addition, the signature verification did not assert all user-supplied public key properties (like `creatorPubkey`, `voterPubkey`, `raterPubkey`, etc.) against the authenticated `X-Public-Key` (`pubKeyHex`), allowing potential identity spoofing within the request body.
**Learning:** In highly customized signature-based auth middlewares, it's critical to ensure all nested paths are covered by wildcard matching or prefixes, and all potential identity properties are validated against the authenticated entity to prevent spoofing in the application layer.
**Prevention:** Use broad path prefixes (`ctx.path.startsWith('/api/marketplace/')`) in middleware authorization blocks and explicitly cross-check all identity fields in request bodies against the authenticated user credential.
